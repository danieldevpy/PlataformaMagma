## O que foi feito

- **Decisão de banco**: o banco de prod é MySQL e o n8n NÃO suporta MySQL (v1+ só
  SQLite/Postgres) → SQLite interno do n8n (volume) em dev E prod; Postgres dedicado
  só se um dia precisar de queue mode. Sem Redis/workers (referência antiga tinha —
  complexidade desnecessária aqui).
- **Prod**: serviço `n8n` no `docker-compose.prod.yml` (loopback `127.0.0.1:5678`,
  volume `n8n_data`, healthcheck, `N8N_ENCRYPTION_KEY`/`N8N_DOMAIN` no `.env.prod`);
  bloco do subdomínio `n8n.magmacursosltda.com.br` no `nginx/nginx.conf` (websocket);
  `init-prod.sh` barra placeholder da chave e mostra o n8n nos acessos.
- **Dev**: `plataforma/n8n/docker-compose.dev.yml` (container único, localhost:5678,
  `host.docker.internal` p/ falar com o runserver) + flag `--n8n` no `init-dev.sh`.
- **Docs**: `plataforma/n8n/README.md` (decisões, como rodar, checklist de prod,
  integrações: `N8N_LEAD_WEBHOOK` já existia em `apps/leads/signals.py`).
- **Correção aproveitada**: portas 8000/3000 do compose de prod publicavam aberto
  (divergindo do comentário "só loopback") → corrigidas p/ `127.0.0.1:`.
- Validação: `bash -n` nos scripts OK; `docker compose config` OK nos dois composes.
  Boot real do container não foi exercitado nesta sessão (Daniel interrompeu o
  `up -d` de teste) — primeiro `init-dev.sh --n8n` fará o pull da imagem.
